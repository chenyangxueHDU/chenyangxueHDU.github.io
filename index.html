《Docker-容器与容器云》读书笔记-docker网络基础

发表于 2017-12-13 | 分类于 Docker

引言

在Docker中，网络已经被抽离成一个单独的库——libnetwork，Dokcer daemon通过调用libnetwork库的API来管理容器网络。为了标准化网络驱动开发和支持多种网络驱动，Docker在libnetwork中使用CNM（Container Network Model）定义容器的虚拟化网络模型。

CNM

在CNM中主要有三种组件——Sandbox(沙盒)、Endpoint(端点)以及Network（网络）。

Sandbox

一个沙盒包括了一个容器网络栈的信息。沙盒可以对容器的接口、路由和DNS设置进行管理。一个沙盒可以有 多个 网络的 许多 端点。

Network

网络是一组可以直接互相连通的端点。网络可以有多种实现：bridge, host, ipvlan, macvlan, null, overlay, remote, windows。

Endpoint

端点代表一个沙盒与一个网络之间的逻辑链接。

Bridge

Bridge是Docker网络的默认实现方式。

docker0

安装完Docker之后，宿主机上就会多一个名为docker0的网卡，IP为172.17.0.1。

运行一个容器，并进入容器里面可以看到绑定了一张网卡，IP为172.17.0.2；同时宿主机上多了一个名为veth6a8174b的网卡。之前已经学过两个network namespace之间可以通过veth pair通信，即veth pair的一端是docker0，另一端就是容器中的eth0。

《Docker 容器与容器云》读书笔记--cgroups资源限制

发表于 2017-12-03 | 分类于 Docker

引言

cgroups–control groups，把任务(进程、线程)放到一个组里面同意加以控制，可以限制、记录任务组所使用的物理资源（CPU、Memory、IO），是构建虚拟化管理工具的基石。

作用

  • 资源限制：对人物使用的资源总额进行限制。设定应用使用内存的上线。
  • 优先级分配：通过分配CPU时间片数量及磁盘IO带宽大小，控制任务运行优先级。
  • 资源统计：统计系统的资源使用量，如CPU使用时长，内存用量；适用于计费。
  • 任务控制：对任务执行挂起、恢复等操作。

术语

  • task任务：进程、线程
  • cgroup控制组：按照某种资源控制标准而划分的任务组，包含一个或多个子系统
  • subsystem子系统：一个资源调度控制器。如CPU子系统控制CPU时间分配，内存子系统限制内存使用量
  • hierarchy层级：层级由一系列cgroup树状结构排列而成，每个层级通过绑定对应子系统进行资源控制

子系统

  • cpu：控制任务对CPU的使用
  • cpuacct：生成任务对CPU资源使用情况报告
  • cpuset：为任务分配独立的CPU（多处理器）和内存
  • memory：设定任务对内存使用量的限定，并生产使用报告
  • devices：开启或关闭任务对设备的访问
  • ferrzer：挂起或恢复任务
  • blkio：为块设备设定输入、输出限制

查看子系统

cgroup的实现是文件系统，所以需要mount。

查看cpu子系统下的控制组

看到Docker会在每个子系统的控制组目录下创建一个名为docker的控制组，并且在docker控制组里再为每个容器创建一个以容器ID为名称的容器控制组，容器中所有的进程PID都会写入该控制组的tasks文件中。

以资源开头（cpu.）的文件都是限制这个cgroup的下任务的配置文件；tasks文件保存该cgroup任务的ID。

《Docker 容器与容器云》读书笔记--namespace资源隔离

发表于 2017-12-03 | 分类于 Docker

引言

如何实现一个资源隔离的容器？首先隔离文件系统；接着为了通信，容器必须有独立的IP、端口、路由等，也就是隔离网络；容器需要一个独立的主机名以便在网络中标识自己，隔离主机名和域名；进程间通信IPC也需要隔离；权限也需要隔离；最后需要有自己的PID。

linux提供了6种隔离机制。

namespace 系统调用参数 隔离内容
UTS CLONE_NEWUTS 主机名与域名
IPC CLONE_NEWIPC 信号量、消息队列和共享内存
PID CLONE_NEWPID 进程编号
Network CLONE_NEWNET 网络设备、网络栈、端口等
Mount CLONE_NEWNS 挂载点（文件系统）
User CLONE_NEWUSER 用户和用户组

namespace API

clone()

创建新进程和namespace

setns()

加入一个已经存在的namespace

unshare()

在原先进程上进行namespace隔离；目前Docker暂未使用该系统调用。

PID namespace

对PID重新编号，即不同namespace下的进程可以有相同的PID。内核为PID namespace维护了一个树状结构，顶层是系统初始化创建的，root namespace。它创建的新PID namespace是child namespace。父节点可以看到子节点中的进程，并通过信号等IPC方式对子节点中的进程产生影响；反之不可以。

init进程

PID为1的进程是init，负责处理“孤儿”子进程（因父进程错误），回收资源结束进程。PID namespace维护树状结构是为了有利于系统的资源监控和回收。

信号屏蔽

为了防止init进程被误伤，init进程会对同一PID namespace下的进程发送的信号进行屏蔽。
如果父节点PID namespace对子节点发送SIGKILL和SIGSTOP信号，那么子节点init会强制执行。一旦init进程被销毁，那么同一PID namespace下的其他进程都会收到SIGKILL。

mount namespace

通过隔离文件系统挂载点对隔离文件系统提供支持，是第一个linux namespace，所以标识位很特殊——CLONE_NEWNS。创建mount namespace时，会复制当前文件结构给新的namespace。

挂载关系

  • 共享关系: 如果两个挂载对象具有共享关系，那么一个挂在对象中的挂在时间会传播到另一个挂在对象，反之亦然。各个namespace之间的目录相互影响。
  • 从属关系: 只有从属对象接受挂在事件。mount namespace中的目录发生变化，child namespace会受到影响；child namespace目录发生变化，mount namespace不会受到影响。
  • 私有关系: 各个namespace完全隔离。
  • 不可绑定挂载： 不允许其他mount namespace挂挂载。

network namespace

一个物理的网络设备最多存在于一个namespace中，通过穿件veth pair(虚拟网络设备对：有两端，类似管道，如果数据从一端传入另一端也可以接收到，反之亦然)在不同的network namespace之间创建通道，以达到通信目的。
在veth pair创建完成之前，docker daemon通过pipe与容器的init进程通信。
